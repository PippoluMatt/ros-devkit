# Extend shared libraries and migrate remaining skills to skill-local library packages

ADR 0001 established `utils/` and `cmake_lib/` as shared library packages and `ros2_control_pluginize_lib/` as the first skill-local library package. Two skills still carried their scripts inside their own folders as flat files: `description-scaffold/scripts/` (three files using `importlib` path-loading for tests) and `gazebo-simulation/scripts/` (a 1585-line monolith). Both duplicated `find_package_name`, `robot_name_from_package`, `_cmake_without_comments`, and `_installed_share_directories` — the same class of divergence ADR 0001 addressed.

Created `package_xml_lib/` as a third shared library package for `package.xml` read and mutation operations (`read_package_name`, `read_dependencies`, `robot_name_from_package`, `ensure_dependencies`, `ensure_exec_depends`, plus a standalone CLI). Absorbed `_cmake_without_comments` (already existed as `cmake_lib.parsing.strip_cmake_comments`) and `_installed_share_directories` (new `cmake_lib.parsing.installed_share_directories`) into `cmake_lib`. Migrated `description-scaffold` and `gazebo-simulation` scripts into `description_scaffold_lib/` and `gazebo_simulation_lib/` under `scripts/`, following the module-by-concern pattern established by `ros2_control_pluginize_lib/`. Updated `registry.py` to point at the new `__main__.py` entry points.

The alternative was leaving each skill's scripts in its own folder. This was rejected because the duplicated helpers had already started diverging (`find_package_name` had three copies with slightly different fallback logic; `_installed_share_directories` had two copies with different comment-stripping), and the `importlib` path-loading test pattern was fragile and broke when any script moved.
